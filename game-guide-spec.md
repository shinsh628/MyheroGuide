## B. 전체 구조

### B-1. 직업 목록 (5종)

`전사` / `헌터` / `법사` / `사제` / `도적`

### B-2. 빌드 구조

```
빌드 식별자 = (무기종류명, 직업명, 빌드번호)

- 무기종류명: n종 중 하나
- 직업명: 5종 중 하나
- 빌드번호: 같은 (무기, 직업) 조합 내에서 선형 인덱스 (0, 1, 2, ...)

빌드번호는 해당 빌드의 이름(편집자 입력)만 결정
그 외 탭 데이터 구조에 영향 없음
```

예시 — 무기 a, 직업 전사에 빌드가 3개인 경우:
```
a전사 #0 "딜특화"
a전사 #1 "생존특화"
a전사 #2 "파티버프"
```

### B-3. 사용자 흐름

```
인트로 페이지
  └─ 직업 선택
       └─ 무기 선택
            └─ 빌드 인덱스 선택 (같은 무기+직업 내 빌드명 목록)
                 └─ 공략 페이지
                      ├─ 탭: 장비
                      ├─ 탭: 무기
                      ├─ 탭: 룬
                      ├─ 탭: 스킬
                      ├─ 탭: 카드
                      ├─ 탭: 인형
                      ├─ 탭: 점성술
                      ├─ 탭: 주문
                      ├─ 탭: 펫
                      ├─ 탭: 보석
                      ├─ 탭: 소울파워
                      ├─ 탭: 구슬
                      └─ 탭: 은하장비
```

## C. 빌드 데이터 스키마

> 각 탭의 데이터 구조를 정의합니다.  
> 공통 DB(장비 목록, 속성 풀 등)와 빌드별 데이터로 분리합니다.
> 
> 모든 빌드 JSON은 공통적으로 아래 메타 필드를 포함합니다:
> ```js
> {
>   buildId: "sword_전사_0",   // (무기종류_직업_빌드번호)
>   buildName: "딜특화",        // 편집자 입력
>   status: "pending" | "approved",  // 공략 사이트 표시 여부
>   ...                         // 각 탭 데이터
> }
> ```

### 1. 장비

#### 1-1. 시스템 개요

- 슬롯 6종: `투구` / `망토` / `상의` / `목걸이` / `반지` / `신발`
- 각 장비는 고유 이름을 가지며, 이름으로 이미지·고유설명·세트소속이 자동 매칭됨
- 혼돈 세트 구조
  - 각 장비는 하나의 혼돈 세트 이름이 고정되어 있음 (장비 고유 속성, 선택지 아님)
  - 게임 설계상 6부위를 3+3으로 나눠 각 그룹이 동일 세트 → 슬롯당 세트레벨 1~3, 세트당 최대 9렙
  - 6부위 풀장착 시 자동으로 9+9렙 달성 (공략에서 별도 명시 불필요)
- 속성 수치: 전체 속성 풀에서 랜덤 획득, 레벨 1~11 랜덤
- 옵션 스킬: 전체 옵션스킬 풀에서 랜덤 획득, 레벨 1~11 랜덤
- 장비 고유설명: 장비마다 고정, 변하지 않음

#### 1-2. 공통 DB 구조

```js
// 장비 목록 DB
equipment: [
  {
    id: "equip_001",
    name: "장비이름",
    image: "경로 또는 URL",
    description: "고유 설명 텍스트",
    chaosSet: "세트이름" // 장비 고유의 고정값, 선택지 아님
  },
  ...
]

// 속성 풀 (에디터 드롭다운용, 사용자에게 노출 안 됨)
statPool: ["체력", "공격력", "치명타피해", ...]

// 옵션스킬 풀 (에디터 드롭다운용, 사용자에게 노출 안 됨)
optionPool: ["옵션스킬A", "옵션스킬B", ...]
```

#### 1-3. 빌드별 데이터 구조

```js
equipment: [
  {
    slot: "투구",           // 슬롯명 (고정)
    equipId: "equip_001",  // 장비 DB 참조 → 이름/이미지/설명/세트 자동매칭
    requiredStats: [
      "공격력",
      "치명타피해"
    ],
    requiredOptions: [
      { name: "옵션스킬A", minLevel: 5, priority: 1 },
      { name: "옵션스킬B", minLevel: 3, priority: 2 }
    ]
  },
  // ... 나머지 5슬롯
]
```

#### 1-4. 공략 페이지 표시 항목

| 표시 항목 | 데이터 출처 |
|-----------|------------|
| 슬롯명 | 고정 |
| 장비 이름 | 장비 DB |
| 장비 이미지 | 장비 DB |
| 장비 고유설명 | 장비 DB |
| 세트 이름 및 레벨 | 장비 DB (고정값, 자동표시) |
| 필수 속성 목록 | 빌드 데이터 |
| 필수 옵션스킬 + 최소레벨 + 우선순위 | 빌드 데이터 |

### 2. 무기

#### 2-1. 시스템 개요

- 무기 탭은 **직업무기 1개 + 한정무기 2개**로 구성
- 직업무기
  - 빌드 식별자 `(무기종류명, 직업명)`에 의해 자동 확정 → 에디터에서 선택 불필요
  - 랜덤 속성 3개 보유, 수치 강화 가능 → 공략에서 **속성 우선순위**만 안내
- 한정무기
  - 직업/빌드에 무관하게 모든 한정무기 중 선택 가능
  - 공략에서 추천 한정무기 2종을 안내해야 함
  - 랜덤 속성 3개 + 각성속성 1개 보유, 수치 강화 가능 → **속성 우선순위 4개** 안내
- 속성 풀: 모든 무기(직업무기/한정무기)가 동일한 풀 공유

#### 2-2. 공통 DB 구조

```js
// 직업무기 DB
classWeapons: [
  {
    id: "cw_001",
    weaponType: "무기종류명",  // 빌드 식별자의 무기종류와 연결
    class: "전사",             // 직업 귀속
    name: "직업무기 고유명",
    image: "경로 또는 URL",
    description: "고유 설명 텍스트"
  },
  ...
]

// 한정무기 DB
limitedWeapons: [
  {
    id: "lw_001",
    name: "한정무기 고유명",
    image: "경로 또는 URL",
    description: "고유 설명 텍스트"
  },
  ...
]

// 무기 속성 풀 (직업무기/한정무기 공통, 에디터 드롭다운용)
weaponStatPool: ["속성A", "속성B", ...]
```

#### 2-3. 빌드별 데이터 구조

```js
weapon: {
  classWeapon: {
    // weaponId는 빌드 식별자에서 자동결정, 별도 저장 불필요
    statPriority: ["속성A", "속성B", "속성C"]  // 1순위 → 3순위
  },
  limitedWeapons: [
    {
      weaponId: "lw_001",
      statPriority: ["속성A", "속성B", "속성C", "각성속성D"]  // 1순위 → 4순위
    },
    {
      weaponId: "lw_002",
      statPriority: ["속성A", "속성B", "속성C", "각성속성D"]
    }
  ]
}
```

#### 2-4. 공략 페이지 표시 항목

| 무기 종류 | 표시 항목 | 데이터 출처 |
|-----------|-----------|------------|
| 직업무기 | 무기 이름, 이미지, 고유설명 | 직업무기 DB (자동매칭) |
| 직업무기 | 속성 우선순위 (3개) | 빌드 데이터 |
| 한정무기 × 2 | 무기 이름, 이미지, 고유설명 | 한정무기 DB (선택 매칭) |
| 한정무기 × 2 | 속성 우선순위 (3개 + 각성 1개) | 빌드 데이터 |

### 3. 룬

#### 3-1. 시스템 개요

- 소켓 6종: `투구` / `망토` / `상의` / `목걸이` / `반지` / `신발` (장비 슬롯명과 동일)
- 각 소켓에 **룬 코어 1개 + 룬 결정 1개** 장착
- 총 12개 슬롯 (6소켓 × 2종)
- 룬 코어와 룬 결정은 각각 별도의 풀에서 선택
- 공략에서는 소켓별로 어떤 코어/결정을 선택해야 하는지 안내

#### 3-2. 공통 DB 구조

```js
// 룬 코어 풀 (에디터 드롭다운용)
runeCorePool: ["코어A", "코어B", ...]

// 룬 결정 풀 (에디터 드롭다운용)
runeCrystalPool: ["결정A", "결정B", ...]
```

#### 3-3. 빌드별 데이터 구조

```js
rune: [
  { slot: "투구",  core: "코어A", crystal: "결정B" },
  { slot: "망토",  core: "코어C", crystal: "결정A" },
  { slot: "상의",  core: "코어B", crystal: "결정C" },
  { slot: "목걸이", core: "코어A", crystal: "결정D" },
  { slot: "반지",  core: "코어D", crystal: "결정A" },
  { slot: "신발",  core: "코어B", crystal: "결정B" }
]
```

#### 3-4. 공략 페이지 표시 항목

| 표시 항목 | 데이터 출처 |
|-----------|------------|
| 소켓명 (6개) | 고정 |
| 소켓별 룬 코어 | 빌드 데이터 |
| 소켓별 룬 결정 | 빌드 데이터 |

### 5. 카드

#### 5-1. 시스템 개요

- 장착 슬롯: **무기 3개 / 방어구 4개 / 장신구 2개** (총 9슬롯, 미장착 가능)
- 카드는 부위별 풀(무기/방어구/장신구) 3개로 분리 관리
- 카드 이름으로 이미지·설명·부위·코스트 자동 매칭
- 주속성·보조속성은 부위와 무관한 별도 풀, 편집자가 드롭다운으로 우선순위 선택

#### 5-2. 공통 DB 구조

```js
// 카드 DB (부위별 3풀)
cards: {
  weapon: [
    { id: "cw_001", name: "카드명", image: "...", description: "...", cost: 3 },
    ...
  ],
  armor: [
    { id: "ca_001", name: "카드명", image: "...", description: "...", cost: 2 },
    ...
  ],
  accessory: [
    { id: "cc_001", name: "카드명", image: "...", description: "...", cost: 4 },
    ...
  ]
}

// 주속성 풀 (에디터 드롭다운용)
cardMainStatPool: ["주속성A", "주속성B", ...]

// 보조속성 풀 (에디터 드롭다운용)
cardSubStatPool: ["보조속성A", "보조속성B", ...]
```

#### 5-3. 빌드별 데이터 구조

```js
card: {
  weapon: [
    { cardId: "cw_001", mainStatPriority: ["주속성A", "주속성B"], subStatPriority: ["보조속성A", "보조속성B"] },
    { cardId: "cw_002", mainStatPriority: [...], subStatPriority: [...] },
    { cardId: null }  // 미장착 슬롯
  ],
  armor: [
    { cardId: "ca_001", mainStatPriority: [...], subStatPriority: [...] },
    { cardId: "ca_002", mainStatPriority: [...], subStatPriority: [...] },
    { cardId: null },
    { cardId: null }
  ],
  accessory: [
    { cardId: "cc_001", mainStatPriority: [...], subStatPriority: [...] },
    { cardId: null }
  ]
}
```

#### 5-4. 공략 페이지 표시 항목

| 슬롯 | 표시 항목 | 데이터 출처 |
|------|-----------|------------|
| 무기 × 3 | 카드 이름, 이미지, 설명, 코스트 | 카드 DB (자동매칭) |
| 무기 × 3 | 주속성 우선순위, 보조속성 우선순위 | 빌드 데이터 |
| 방어구 × 4 | 위와 동일 | 카드 DB + 빌드 데이터 |
| 장신구 × 2 | 위와 동일 | 카드 DB + 빌드 데이터 |
| 미장착 슬롯 | 빈 슬롯으로 표시 | - |

### 6. 인형

#### 6-1. 시스템 개요

- 편집자가 업로드한 이미지 1장만 표시
- DB나 별도 구조 없음, 빌드별 이미지 1개만 저장

#### 6-2. 빌드별 데이터 구조

```js
doll: {
  image: "경로 또는 URL"  // 편집자가 업로드한 이미지
}
```

#### 6-3. 공략 페이지 표시 항목

| 표시 항목 | 데이터 출처 |
|-----------|------------|
| 이미지 1장 | 빌드 데이터 (편집자 업로드) |

### 7. 점성술

#### 7-1. 시스템 개요

- 슬롯 3개, 단일 공통풀에서 선택
- 이름으로 이미지·설명 자동 매칭

#### 7-2. 공통 DB 구조

```js
astrology: [
  { id: "as_001", name: "점성술명", image: "...", description: "..." },
  ...
]
```

#### 7-3. 빌드별 데이터 구조

```js
astrology: ["as_001", "as_003", "as_007"]  // 슬롯 3개
```

#### 7-4. 공략 페이지 표시 항목

| 표시 항목 | 데이터 출처 |
|-----------|------------|
| 점성술 이름, 이미지, 설명 (3개) | 점성술 DB (자동매칭) |

---

### 8. 주문

#### 8-1. 시스템 개요

- 슬롯 3개, 단일 공통풀에서 선택
- 이름으로 설명 자동 매칭 (이미지 없음)

#### 8-2. 공통 DB 구조

```js
spells: [
  { id: "sp_001", name: "주문명", description: "..." },
  ...
]
```

## E. 에디터

### E-1. UI 구조

- 공략 페이지와 동일한 레이아웃
- 차이점: 각 항목이 드롭다운/입력 필드로 편집 가능한 상태로 렌더링
- 즉 편집하면서 실제 공략 페이지가 어떻게 보일지 실시간 확인 가능

### E-2. 권한별 제출 동작

```
화이트리스트 편집자
  └─ 제출 버튼
       → JSON 생성 (status: "approved")
       → Netlify Function → GitHub 푸시
       → 즉시 공략 사이트에 표시

일반 편집자
  └─ 제출 버튼 (5시간 내 최대 3회 제한)
       → JSON 생성 (status: "pending")
       → Netlify Function → GitHub 푸시
       → 공략 사이트에 미표시
       → 관리자 탭에서 pending 목록 확인
       → 승인 버튼 → status "approved"로 변경 → 자동 배포 → 공략 페이지 표시
```

### E-3. 제출 제한

| 구분 | 제한 |
|------|------|
| 화이트리스트 | 제한 없음 |
| 일반 편집자 | 5시간 내 3회 |

### E-5. 화이트리스트 인증 방식

- 제출 시 비밀번호 입력란 추가
- 비밀번호 일치 여부는 Netlify Function 서버사이드에서 검증
- 비밀번호는 Netlify 환경변수에 저장 (코드 노출 없음)
- 비밀번호 맞으면 화이트리스트 → 즉시 푸시, 틀리면 일반 편집자 → 대기열

### E-4. 미결 항목

- 에디터 접근 URL (예: `/editor`)
- 편집자 로그인 방식 (이메일? 닉네임+비밀번호?)
- 승인 대기열 관리 UI (화이트리스트가 승인하는 화면)
- 일반 편집자 제한 초과 시 안내 문구

## F. 배포

| 항목 | 내용 |
|------|------|
| 플랫폼 | Netlify |
| 비용 | 무료 티어 |
| 배포 방식 | GitHub 레포 연동 → main 브랜치 푸시 시 자동 배포 |
| 공략 추가 시 | 에디터에서 작성 후 제출 → 자동 반영 또는 승인 후 반영 |

### F-1. 편집자 권한 구조

```
편집자 (웹 에디터 접속)
  └─ 공략 작성 후 제출
       ├─ 일반 편집자 → Netlify Function → GitHub PR 생성
       │                                      → 관리자 승인(Merge)
       │                                           → 자동 배포
       └─ 화이트리스트 편집자 → Netlify Function → GitHub 직접 푸시 → 자동 배포
```

### F-2. 보안

- GitHub Personal Access Token은 Netlify 환경변수에 저장 (코드 노출 없음)
- 화이트리스트 목록은 서버사이드(Netlify Function)에서 관리
- 편집자는 GitHub 계정 불필요, 웹 에디터만 사용
