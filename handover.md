# Claude Code 핸드오버 문서

> 이 문서는 `game-guide-spec.md`와 함께 읽어야 합니다.  
> 스펙 문서에 데이터 구조와 기획 의도가 상세히 기술되어 있습니다.  
> 이 문서는 구현 순서, 주의사항, 의사결정 기준을 안내합니다.

---

## 1. 프로젝트 개요

모바일 핵앤슬래시 게임의 빌드 공략 사이트.  
HTML/CSS/JS 정적 사이트 + Netlify Functions 백엔드.  
GitHub 레포 → Netlify 자동 배포.

**운영자가 에디터로 공략을 작성 → JSON 저장 → 사이트에 렌더링**하는 구조가 핵심.  
코드의 확장성과 에디터의 사용 편의성을 최우선으로 설계할 것.

---

## 2. 구현 순서

아래 순서대로 구현. 각 단계 완료 후 운영자와 확인하고 다음 단계 진행.

### Phase 1. 프로젝트 초기 세팅
- 폴더/파일 구조 설계 및 생성
- DB 파일 구조 작성 (`/data/db.js` 또는 JSON)
- 빌드 데이터 샘플 1개 작성 (더미 데이터)
- Netlify 설정 파일 (`netlify.toml`)

### Phase 2. 공략 페이지 (읽기 전용)
- 인트로 → 직업 선택 → 무기 선택 → 빌드 인덱스 선택 흐름
- 공략 페이지 레이아웃 + 13개 탭 UI
- 각 탭별 렌더링 로직 (DB 데이터 → 화면 출력)
- 모바일 반응형 적용

### Phase 3. 에디터
- 공략 페이지와 동일한 레이아웃에서 드롭다운/입력 편집 가능
- 각 탭별 편집 UI (드롭다운은 DB 풀 기반)
- 비밀번호 입력란 (화이트리스트 여부 결정)
- 제출 버튼 → Netlify Function 호출

### Phase 4. Netlify Functions
- 화이트리스트 검증 (환경변수 비밀번호 비교)
- GitHub API로 JSON 파일 푸시
  - 화이트리스트: `status: "approved"` → 즉시 푸시
  - 일반 편집자: `status: "pending"` → 푸시 후 미표시
- 일반 편집자 제출 횟수 제한 (5시간 내 3회)

### Phase 5. 관리자 화면
- `/admin` 경로
- `status: "pending"` 빌드 목록 표시
- 승인 버튼 → `status: "approved"`로 변경 → 푸시

## 4. 핵심 설계 원칙

### 4-1. DB와 빌드 데이터 분리
- **공통 DB** (`data/db.js`): 장비 목록, 카드 목록, 속성 풀 등 → 사이트 전체 공유
- **빌드 데이터** (`data/builds/*.json`): 빌드별 독립 파일 → 에디터가 생성

이 분리가 깨지면 확장성이 무너짐. 빌드 데이터에 원본 텍스트를 중복 저장하지 말 것.  
빌드 데이터는 항상 DB의 `id`를 참조하고, 렌더링 시점에 매칭할 것.

### 4-2. 빌드 식별자
```
buildId = "{무기종류}_{직업}_{빌드번호}"
예) "sword_전사_0", "sword_전사_1", "bow_헌터_0"
```

### 4-3. 빌드 JSON 공통 메타 필드
```js
{
  buildId: "sword_전사_0",
  buildName: "딜특화",           // 편집자 입력
  status: "pending" | "approved", // 공략 사이트 표시 여부
  // 이하 각 탭 데이터
  equipment: [...],
  weapon: {...},
  rune: [...],
  ...
}
```

### 4-4. 탭 13개 중 미작성 항목
- `13. 은하장비`: 나중에 추가 예정. 탭 슬롯만 예약해두고 "준비 중" 표시
- 나머지 12개 탭은 스펙 문서 `C` 섹션 참고

### 4-5. 이미지 처리
- 모든 이미지는 URL 또는 상대경로로 DB에 저장
- 에디터에서 이미지 URL을 직접 입력하는 방식으로 구현
- 파일 업로드 기능은 현재 스펙에 없음

## 5. 구현 시 주의사항

- **모바일 우선**: 주 접속 경로가 모바일. 레이아웃은 모바일 기준으로 먼저 설계하고 데스크탑 확장
- **탭이 많음**: 공략 페이지에 탭 13개. 모바일에서 탭 UI가 깨지지 않게 스크롤 또는 드롭다운 방식 고려
- **에디터 = 공략 페이지**: 에디터 UI는 공략 페이지와 최대한 동일하게 유지. 별도 레이아웃 만들지 말 것
- **풀 목록은 나중에 채움**: DB의 속성 풀, 옵션스킬 풀 등 실제 항목은 운영자가 각 탭 작업 시 제공. 지금은 샘플 몇 개만 넣어둘 것
- **Netlify Function 토큰**: GitHub Personal Access Token과 화이트리스트 비밀번호는 절대 코드에 하드코딩하지 말 것. 환경변수(`process.env.GITHUB_TOKEN`, `process.env.WHITELIST_PASSWORD`)로만 접근

## 6. 운영자와 확인이 필요한 시점

아래 시점에 반드시 운영자에게 확인 후 진행:

1. **Phase 1 완료 후**: 폴더 구조와 DB 스키마 확인
2. **Phase 2 완료 후**: 공략 페이지 디자인 및 탭 UI 확인
3. **Phase 3 완료 후**: 에디터 사용성 확인
4. **각 탭 구현 시**: 해당 탭의 풀 목록(속성, 옵션스킬 등) 운영자에게 요청
5. **은하장비 탭**: 운영자가 별도로 스펙 제공 예정, 그 전까지 "준비 중" 처리
